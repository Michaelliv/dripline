Warehouse Mode

Turn dripline into a lightweight data warehouse on top of any S3-compatible bucket. Same plugins, same SQL — but data accumulates over time and queries read parquet, not live APIs.

Plugin APIs ──→ dripline run ──→ raw/ (append-only parquet) │ dripline compact │ ▼ curated/ (deduped, partitioned parquet) │ dripline query --remote │ ▼ DuckDB SQL

Three commands, one bucket. No scheduler service, no catalog — the bucket is the source of truth and the coordination layer.

Setup

$ export R2_KEY_ID=...
$ export R2_SECRET=...

# Point at a bucket
$ dripline remote set https://<account>.r2.cloudflarestorage.com \
    --bucket my-warehouse --prefix prod --secret-type S3 \
    --access-key-env R2_KEY_ID --secret-key-env R2_SECRET

# Define lanes
$ dripline lane add orders \
    --table orders --params org_id=abc123 \
    --interval 15m

# Sync, compact, query
$ dripline run
$ dripline compact
$ dripline query --remote "SELECT COUNT(*) FROM orders"

Lanes

A lane is a named group of tables synced together on a schedule. Each table can have its own params — key column values passed to the plugin.

OptionDescription
intervalMinimum time between runs. The lane's lease holds for this duration after completion.
maxRuntimeOptional hard cap. Useful for long-running backfills that should yield to other lanes.

The same table can appear in multiple lanes with different params (e.g. different orgs). Tables within a lane sync sequentially; lanes are independent.

Leases

Workers coordinate via JSON files in _leases/. When dripline run starts a lane, it writes a lease with its holder ID and expiry. Other workers skip that lane. After completion, the lease holds for the lane's interval — a cooldown that prevents unnecessary re-runs.

Run dripline run from multiple machines on a cron — they divide lanes via contention. No leader election, no rebalancing.

Releasing stale leases

# Release lease, keep cursor (safe default)
$ dripline lane reset orders --yes

# Release lease AND delete cursor (full rebackfill)
$ dripline lane reset orders --hard --yes

Compaction

Compact merges raw/ into curated/. Deduplicates by primary key (latest cursor wins), partitions by key columns, and cleans up consumed raw files.

Incremental by default

When raw data only touches 2 out of 8,000 partitions, compact reads those 2 curated partitions — not all 8,000.

The algorithm:

1. List raw files. If none → skip.

2. Read raw parquet, extract distinct partition values as literals.

3. Read only the matching curated partitions. DuckDB's hive pruning skips non-matching files at the directory level.

4. Union raw + affected curated, deduplicate by PK via ROW_NUMBER.

5. Write back with PARTITION_BY. Only affected partition files are overwritten.

6. Delete consumed raw files.

This turns a 66-minute full recompaction into a 5-second incremental compact for a daily sync.

Known limitation

If a row's partition column value changes between syncs (e.g. business_date corrected), the old partition retains the stale copy. Incremental compact only reads partitions present in the new raw data. In practice this is rare — partition columns like org_id are effectively immutable.

Sync Progress

During long-running syncs, the engine fires progress events every 10,000 rows:

ℹ [detail] github_issues: 50,000 rows (6.8/s cursor=2025-08-28)

Available via the SDK:

await dl.sync(params, (event) => {
  console.log(`${event.table}: ${event.rowsInserted} rows`);
});

Bucket Layout

<prefix>/ ├── _leases/ │ └── lane-<name>.json lease + cooldown timer ├── _state/ │ └── <lane>/_dripline_sync.parquet cursor metadata ├── _manifests/ │ └── <table>.json file index + partition stats ├── raw/ │ └── <table>/lane=<lane>/run=<id>.parquet append-only └── curated/ └── <table>/<partition>=<val>/data_0.parquet deduplicated

Production

# One cron line is all you need:
*/15 * * * * cd /path/to/project && source .env && dripline run && dripline compact
ConcernAnswer
IntervalSet cron to your shortest lane interval. Longer lanes skip automatically.
ScalingAdd more cron entries on different machines. Lease contention divides work.
CredentialsUse --access-key-env / --secret-key-env to keep secrets out of config.
MonitoringExit 0 on success, 1 on failure. Logs include row counts, durations, cursors.
BackfillSet initialCursor in the plugin table definition. First run backfills from there.
CostR2: no egress, ~$0.015/GB/month storage, $4.50/M PUTs. Pennies for typical warehouses.
